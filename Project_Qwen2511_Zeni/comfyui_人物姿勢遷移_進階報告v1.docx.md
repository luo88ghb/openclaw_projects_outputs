**ComfyUI 結合 OpenPose 與 IP-Adapter 的人物姿勢遷移專案技術報告（第一版）**

**文件定位**

本文件定義一個以本地部署為核心、避免付費 API 的「人物／角色姿勢遷移專案」第一版技術架構，目標是在 ComfyUI 內以可維護、可逐步自動化的方式，將圖像 1 的姿勢條件遷移到圖像 2 的目標主體，同時盡量保留圖像 2 的身份特徵、局部外觀與整體可辨識性。\[cite:1\]\[cite:15\]

本專案的合理技術基線，不應被定義為單一步驟的「直接換姿勢」，而應拆成三個受控子任務：姿勢擷取、身份／特徵保留、以及細節重建與局部修正；這是因為 OpenPose 類條件控制擅長提供姿勢骨架，但身份一致性通常還需依賴 IP-Adapter FaceID、LoRA 或多階段重繪流程補足。\[cite:1\]\[cite:15\]

**專案目標定義**

**核心目標**

本專案要解決的不是一般文生圖，而是「條件受控生成」：從來源圖像擷取姿勢條件，並把該條件施加到另一個目標主體上，使輸出結果同時滿足姿勢相似、主體可辨識、構圖合理與畫面品質可接受等四個要求。\[cite:1\]

在工程定義上，第一版專案應優先聚焦於「單主體、單張輸入、單張輸出」場景，先建立穩定 SOP，再擴展到批次自動化、多角色、跨物種與高解析兩階段修復流程。\[cite:1\]

**成功條件**

第一版若要被視為成功，至少要達成下列條件：

* 來源圖像可穩定產出可用的姿勢條件圖或關鍵點資訊。\[cite:1\]

* 目標圖像主體的核心外觀特徵可在多次採樣下保持中高度一致，而不是每次都變成不同角色。\[cite:15\]

* 在人物對人物的情境中，輸出姿勢應明顯對齊來源骨架，且肢體結構不能頻繁崩壞。\[cite:1\]

* 在高難度案例，例如「人類姿勢遷移到貓」時，系統應標記為高風險模式，而非承諾精確一比一映射。\[cite:1\]

**系統架構**

**整體邏輯分層**

建議將系統拆成五層：輸入層、條件抽取層、條件融合層、生成／修復層、驗證層。這樣的拆法有助於後續把 ComfyUI 工作流從「手動試圖調參」升級成「可維護的節點化流水線」。\[cite:1\]

| 層級 | 功能 | 建議工具／模組 | 備註 |
| :---- | :---- | :---- | :---- |
| 輸入層 | 載入圖像 1 與圖像 2 | ComfyUI Load Image | 單主體為第一優先 |
| 條件抽取層 | 擷取姿勢、臉部、必要局部資訊 | OpenPose、InsightFace 流程 | 臉部識別只適用人臉案例 \[cite:15\] |
| 條件融合層 | 把姿勢條件與身份條件合成 | ControlNet、IP-Adapter FaceID | 姿勢與身份需分開控制 \[cite:1\]\[cite:15\] |
| 生成／修復層 | 先出結構，再做高解析與局部修補 | KSampler、VAE、Latent Upscale、局部重繪 | 兩階段流程較穩定 \[cite:1\] |
| 驗證層 | 檢查姿勢、身份、破圖與失真 | 人工檢查 \+ 規則化評估 | 第一版以人工檢核為主 |

**兩條主工作線**

第一版不建議只做一條工作流，而是並行定義兩條主線：一條是「人物對人物精準遷移線」，另一條是「跨物種／角色創意轉譯線」。這兩條線雖可共用 OpenPose 與基礎生成節點，但驗證標準與風險管理完全不同。\[cite:1\]

人物對人物線應追求骨架對齊與身份穩定；跨物種線則應改以「動作語意接近、主體可辨識、畫面合理」為優先，因為 OpenPose 的人體關鍵點定義本身是為人體設計，不保證能對非人類主體提供等價控制。\[cite:1\]

**模型與節點選型**

**必要模組**

第一版最核心的技術組合是：ComfyUI 作為節點編排介面、OpenPose 對應的 ControlNet 作為姿勢條件控制、Stable Diffusion 基底模型作為生成器，以及 IP-Adapter FaceID 作為人臉身份一致性補強模組。\[cite:1\]\[cite:15\]

ComfyUI 官方 Pose ControlNet 範例明確展示了以 Pose ControlNet 進行兩階段生成的做法，包含載入 checkpoint、VAE、OpenPose 對應 ControlNet 模型、第一階段產生基礎姿勢結構，以及第二階段以較低 denoise 做高解析與細節優化。\[cite:1\]

**模組角色說明**

| 模組 | 任務定位 | 適用場景 | 限制 |
| :---- | :---- | :---- | :---- |
| OpenPose / Pose ControlNet | 提供身體姿勢、臉部、手部等結構條件 | 人物動作遷移 | 對跨物種不保證等價有效 \[cite:1\] |
| Stable Diffusion 基底模型 | 生成主畫面 | 全流程皆需 | 模型風格會影響身份穩定與肢體品質 \[cite:1\] |
| VAE | 解碼與畫質穩定 | 全流程皆需 | 與 checkpoint 搭配不當會影響品質 \[cite:1\] |
| IP-Adapter FaceID | 保留人臉身份特徵 | 人物臉部一致性 | 主要依賴人臉嵌入，非人臉主體不適用 \[cite:15\] |
| FaceID Plus / PlusV2 | 同時利用 face ID 與臉部結構資訊 | 高相似度人物遷移 | 仍有一致性上限 \[cite:15\] |
| LoRA | 強化角色特徵或特定風格 | 重複主角專案 | 需額外訓練或蒐集素材 |

**選型結論**

第一版建議以 SD1.5 生態作為實作起點，而不是一開始就追求最複雜的多模型混合；原因是 ComfyUI 官方 Pose ControlNet 教學與 IP-Adapter FaceID 的公開使用示例，對 SD1.5 體系提供了較成熟、文件較完整的基礎參照。\[cite:1\]\[cite:15\]

若專案第一階段的主要對象是「人臉人物」，則優先選 IP-Adapter-FaceID-Plus 或 PlusV2 路線，因為其設計結合同一人臉 ID 嵌入與結構資訊，目的是提升身份一致性與臉部結構保真度。\[cite:15\]

**ComfyUI 工作流設計**

**工作流 A：人物姿勢遷移 SOP**

此 SOP 對應「ComfyUI 結合 OpenPose 與 IP-Adapter 的完整自動化工作流程設計架構指南」，建議以兩階段生成為主幹，並保留局部修補節點作為第三可選階段。\[cite:1\]\[cite:15\]

**A-1. 輸入階段**

* 載入圖像 1：作為姿勢來源圖。\[cite:1\]

* 載入圖像 2：作為目標主體特徵來源圖。

* 判定案例類型：人物對人物、人物對動物、動物對人物、風格角色對人物。

* 若圖像 2 為人物，額外標記是否可抽取清楚正臉，以決定是否啟用 FaceID 分支。\[cite:15\]

**A-2. 條件抽取階段**

* 由圖像 1 取得 OpenPose 姿勢條件圖，必要時包含手部與臉部關鍵點資訊。\[cite:1\]

* 由圖像 2 萃取身份條件；若為人物臉部，使用 InsightFace 類流程取得 face ID embedding，再送入 IP-Adapter FaceID 模組。\[cite:15\]

* 若圖像 2 不是人物臉部主體，則 FaceID 分支停用，改走「參考圖 \+ prompt \+ 局部遮罩」的弱一致性路線。

**A-3. 第一階段生成：結構優先**

* 載入主 checkpoint、VAE 與 Pose ControlNet 模型。\[cite:1\]

* 正向提示詞只描述必要主體、場景與動作語意，不應在第一階段堆疊過多風格詞。

* 將 OpenPose 條件作為高優先結構控制，讓第一張結果先過關於姿勢與基本構圖。\[cite:1\]

* 若為人物案例，同步施加較保守權重的 IP-Adapter FaceID，避免一開始因身份權重過高而破壞肢體結構。\[cite:15\]

**A-4. 第二階段生成：高解析與特徵回補**

ComfyUI 官方範例指出，兩階段流程可先讓第一階段專注於姿勢與結構，再在第二階段用較低 denoise 進行細節與風格優化，這也是本專案第一版最建議採用的主策略。\[cite:1\]

在第二階段，應以上一階段輸出的 latent 或 decode 結果為基礎進行 upscale 與再採樣，並微調 FaceID 或參考圖條件權重，目標是補回臉部、毛髮、服飾紋理與主體辨識特徵，同時不破壞已建立的骨架。\[cite:1\]\[cite:15\]

**A-5. 第三階段：局部修補（選配）**

* 對臉部、手部、足部、尾巴、腳踏車接觸點等高錯誤區域進行遮罩重繪。

* 對跨物種案例，局部修補通常不是可選項，而是必要項，尤其在「四足動物做出人類騎乘姿勢」時更是如此。

* 這一層的本質是把不可靠的一次性全圖生成，轉成可局部迭代的結構化修復流程。

**工作流 B：ComfyUI 進階遷移**

此工作流對應「如何將圖像 2 的特徵精確對齊至圖像 1 的骨架姿勢（含故障排除）」，其核心不是把所有條件同時拉滿，而是建立權重與優先級管理。\[cite:1\]\[cite:15\]

**B-1. 對齊原則**

* 姿勢控制優先於風格控制，否則骨架容易偏離。\[cite:1\]

* 身份控制優先於裝飾性細節控制，否則角色會漂移成陌生人。\[cite:15\]

* 局部結構控制優先於全局寫實感，否則會得到「畫面好看但姿勢不對」的假成功。

**B-2. 精確對齊策略**

* 使用第一階段低到中解析先驗證骨架是否對，再投入第二階段放大。

* 將 FaceID 權重設定為中低起步，再逐步上調，以避免臉部特徵壓過姿勢條件。\[cite:15\]

* 對於目標主體的關鍵辨識元素，例如臉、毛色塊、服裝輪廓、耳型或尾型，應以局部遮罩回補，而非完全依賴一次生成。

* 若圖像 1 與圖像 2 視角差過大，應先將其視為高風險輸入，而不是直接進入自動批次。

**B-3. 故障排除**

| 問題 | 可能原因 | 處理方向 |
| :---- | :---- | :---- |
| 姿勢不像圖像 1 | Pose ControlNet 權重不足，或第二階段 denoise 過高 | 提高姿勢控制權重，降低第二階段破壞性重採樣 \[cite:1\] |
| 臉不像圖像 2 | FaceID 權重不足，或輸入臉部品質差 | 提升 FaceID 參考品質，改用 Plus/PlusV2，並檢查是否成功抽 embedding \[cite:15\] |
| 骨架正確但肢體扭曲 | 基底模型人體結構能力有限 | 改 checkpoint、增加局部重繪、降低一次生成目標解析度 \[cite:1\] |
| 貓變成人形怪異姿勢 | 人體骨架條件直接套到非人體主體 | 改採語意轉譯策略，不要求逐點骨架一致 \[cite:1\] |
| 放大後姿勢跑掉 | 第二階段 denoise 過高 | 以低 denoise 精修，優先保留第一階段結構 \[cite:1\] |

**輸入資料規格**

第一版專案應強制建立輸入檢查規則，否則工作流再好也會被低品質輸入拖垮。

**風險點**

**技術風險**

最大風險不是單一節點故障，而是多個條件彼此競爭：姿勢要準、身份要像、畫面要自然、場景要合理，這些條件在同一張圖上並不總是可同時最大化。\[cite:1\]\[cite:15\]

IP-Adapter FaceID 的官方模型卡已明確指出，模型無法達到完美寫實與完美身份一致性，其泛化能力也受到訓練資料、基底模型與人臉辨識模型限制，因此第一版專案必須把「近似一致」視為工程目標，而不是宣稱絕對精準複製。\[cite:15\]

**任務風險**

跨物種遷移是高風險任務，特別是把人體姿勢直接轉移到貓、狗或其他非人體主體時；此時較合理的技術目標應是保留動作語意與畫面合理性，而不是人體關鍵點對位的嚴格複製，因為 OpenPose 的條件結構原生就是人體導向。\[cite:1\]

此外，FaceID 相關模型卡亦標示其與 InsightFace 預訓練模型相關的使用限制，並且明確說明該模型釋出目的偏向研究用途而非商業用途；若專案未來有商業化需求，必須在模型授權審查上另行處理。\[cite:15\]

**驗證方法**

**驗證維度**

第一版建議先使用「人工主評 \+ 規則化量表」方式驗證，因為在跨模型、跨風格、跨物種條件下，單一自動評分往往不足以反映實際可用性。

建議每張輸出至少以 5 個面向評分：

* 姿勢對齊度：是否明顯呈現來源圖的主要動作結構。\[cite:1\]

* 身份相似度：是否仍能辨認為圖像 2 的主體；人物案例可聚焦臉部與輪廓。\[cite:15\]

* 結構合理度：肢體、手腳、關節、尾巴、接觸面是否自然。

* 場景一致度：主體與背景互動是否合理，例如騎乘、抓握、站立支撐是否成立。

* 可修復性：若失敗，是否能透過局部重繪修回，而不是整張作廢。

**驗證流程**

建議驗證流程如下：

1. 先做 20 組人物對人物案例，建立基準表現。

2. 再做 10 組跨風格案例，測試風格與身份競爭關係。

3. 最後再做 10 組跨物種案例，驗證高風險路線的失敗型態。

4. 每組至少保留 pass1、pass2 與局部修補版本，對照是哪一層解決了問題。

**分階段落地方案**

**Phase 0：可行性驗證**

目標是確認本地 ComfyUI 環境可穩定跑通官方 Pose ControlNet 兩階段流程，並成功安裝與調用對應模型、VAE 與 ControlNet 權重。\[cite:1\]

此階段不追求最終品質，只驗證節點可運行、模型可載入、輸入輸出格式一致，以及基本姿勢控制有效。\[cite:1\]

**Phase 1：人物對人物 MVP**

此階段聚焦人物對人物遷移，並引入 IP-Adapter FaceID，使系統具備姿勢控制與人物身份維持的最低可用能力。\[cite:15\]

交付標準應為：至少在固定類型輸入上，可穩定輸出一批姿勢正確、臉部中高度相似、可重現參數的結果。\[cite:1\]\[cite:15\]

**Phase 2：半自動故障修復**

此階段加入遮罩重繪、局部修補與參數紀錄機制，讓失敗案例不是直接捨棄，而是進入可回修流程。

這一層完成後，專案才真正具備實用價值，因為人物遷移的品質通常不是一次採樣決定，而是來自多階段修正能力。

**Phase 3：跨物種與創意轉譯**

此階段才建議導入「人騎腳踏車 → 貓騎腳踏車」這類創意型高難度任務，並把驗證標準改為動作語意、主體可辨識與畫面合理，而非嚴格人體骨架對位。\[cite:1\]

在這個階段，工作流應從「精準遷移」轉為「受控轉譯」，否則會因目標設定過度理想化而拖慢整體研發進度。\[cite:1\]

**初版結論**

第一版專案最穩健的技術策略，不是追求一步到位的完美角色換姿，而是以 ComfyUI 為節點基座，先建立 OpenPose ControlNet 的結構控制，再以 IP-Adapter FaceID 補足人臉身份一致性，最後用兩階段生成與局部修補把失敗率壓低到可管理範圍。\[cite:1\]\[cite:15\]

若專案以本地、低授權成本、可持續迭代為前提，這條路線具備相當高的可行性；但對跨物種精確遷移、絕對身份保真與商業授權直接可用性，現階段不應做超出公開資料所支持的承諾。\[cite:1\]\[cite:15\]
